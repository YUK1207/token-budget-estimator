Commercial API layer

Preflight API for agent platforms.

A standard for forecasting spend, evaluating budget policy, and generating execution contracts before a coding agent starts work.

View example See plans
API role Decision gate Not a model billing meter. It controls whether an agent run should start, split, pause, or ask for approval.
Primary buyers Teams and platforms CI systems, internal developer tools, agent platforms, and enterprise engineering organizations.

The API standard is small on purpose.

Five endpoints cover the paid surface: forecast the run, compare work queues, evaluate team policy, generate contracts, and calibrate with real outcomes.

POST /v1/forecast

Forecast

Estimate token range, cost range, risk level, and context fit for one agent task.

POST /v1/batch-forecast

Batch

Compare multiple planned tasks before a shared team budget is spent.

POST /v1/policy/evaluate

Policy

Apply ceilings, approval rules, and split thresholds to a forecast result.

POST /v1/contracts

Contract

Generate stop-loss limits for files, commands, logs, retries, and approval gates.

POST /v1/history/calibrate

Calibrate

Feed actual run outcomes back into the estimator to improve future ranges.

How a platform calls it.

The API should feel predictable to developers: send a task, repo profile, budget ceiling, and price assumptions. Receive a decision and a bounded run contract.

Request

Forecast before execution.

Use this when an internal tool, CI job, or agent platform wants to decide whether a task should run now.

{
  "task": "Fix failing tests in authentication",
  "repo_profile": {
    "language": "typescript",
    "file_count": 420,
    "test_framework": "vitest"
  },
  "budget": {
    "max_tokens": 50000,
    "currency": "USD",
    "input_price_per_million": 1.25,
    "output_price_per_million": 10
  },
  "options": {
    "include_contract": true,
    "include_context_plan": true
  }
}
Response

Return a decision, not just a number.

The response gives the caller a usable action: run, split, block, or ask for approval.

{
  "forecast": {
    "token_range": { "low": 8000, "high": 40000 },
    "cost_range": { "low": 0.08, "high": 0.45 },
    "risk": "medium",
    "context_fit": "fits"
  },
  "recommendation": {
    "action": "run_with_budget_contract",
    "reason": "Task is bounded but may need test log inspection."
  },
  "contract": {
    "mode": "bounded_run",
    "max_files_to_read": 20,
    "max_commands": 8,
    "stop_at": "budget_ceiling"
  }
}

Policy is where the product becomes valuable.

Teams do not only need a forecast. They need a repeatable rule for what happens when the forecast is too large, too risky, or outside the approved budget.

Policy rule
allowForecast is below the team ceiling and risk stays acceptable.
splitForecast is too wide, so the task should become discovery first or staged work.
approveForecast exceeds the no-approval threshold or hits a high-risk category.
blockForecast exceeds the hard budget ceiling or violates a team policy.
Policy response 
{
  "decision": "requires_approval",
  "reasons": [
    "Forecast exceeds 50k token approval threshold",
    "Risk level is high"
  ],
  "suggested_next_step": "split_task"
}
